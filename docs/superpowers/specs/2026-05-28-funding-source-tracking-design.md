# Funding-source tracking

**Date:** 2026-05-28
**Scope:** Add a new dimension to every captured transaction: *which
account paid for it.* Four buckets — Credit Card, E-Wallet, Debit/Bank
Transfer, Cash — with specific account/card identity tracked underneath
the bucket. Classified automatically from notification content with
user override.

---

## Problem

Captured transactions today track *who got paid* (merchant) and *what
for* (category, description), but nothing about *which account paid*.
The user can't answer "how much did I spend on credit cards this
month" or "what did I burn through my TnG wallet on" without manually
re-classifying every row.

The information is sitting right there in the notification text — bank
SMS bodies say `card ending 1234`, `A/C ending 0025`, GWallet pushes
say `with HSBC Platinum ••1868`, wallet apps are self-identifying by
package. A classifier can do most of the work; the user just needs an
override surface for the cases where the heuristic guesses wrong (e.g.,
a debit card whose notifications use the credit-card-style bullets
shape).

## Goal

Every transaction is linked to a *FundingSource*. A FundingSource is a
specific identity (a named card, a named account, a named wallet, or
"Cash"), and each FundingSource belongs to one of four kinds:

- `CREDIT_CARD`
- `EWALLET`
- `DEBIT_BANK`  (covers debit cards and bank-transfer / FPX / DuitNow flows)
- `CASH`

The bucket value (`kind`) lives on `FundingSource`, not on
`Transaction`. CSV exports the bucket name; in-app filters operate at
the bucket level too. Specific card identity surfaces in Settings,
the edit sheet, and the Add Manual sheet.

## Non-goals

- Per-card spending stats / breakdown UI in v1. (CSV column + home
  filter is the read surface.)
- Auto-disambiguation of card-vs-debit beyond the regex-shape
  heuristic. User resolves via the change-bucket action.
- Automatic merge of suspected-duplicate FundingSources (e.g. detecting
  that `HSBC card 1234` from GWallet and `HSBC 1234` from HSBC SMS are
  the same card). Manual merge only.
- Bucket-derived category suggestions. Funding source and category
  stay orthogonal.
- Multi-currency funding sources. A card is just a card; currency is
  per-transaction. User can create additional `Cash` rows in Settings
  for cash-MYR vs cash-USD if they want.

## Data model

### New entity `FundingSource`

| field           | type                       | notes |
|-----------------|----------------------------|-------|
| `id`            | Long PK, autogenerated      | |
| `kind`          | enum `CREDIT_CARD` / `EWALLET` / `DEBIT_BANK` / `CASH` | stored as TEXT |
| `displayName`   | String                     | `"HSBC Visa Platinum 1234"`, `"TnG Wallet"`, `"Cash"`, etc. |
| `last4`         | String?                    | nullable — wallets and the seeded `Cash` source have no last-4 |
| `sourceAppHint` | String?                    | package name where first detected; tiebreaker so two banks with same last-4 don't collide |
| `isUserNamed`   | Boolean                    | false until the user renames; UI marks auto-named entries |
| `createdAt`     | Instant                    | |
| `updatedAt`     | Instant                    | used for backup latest-wins reconciliation |

**Unique index** `(sourceAppHint, last4)`. **`kind` is intentionally NOT
in the unique key** — that is what makes the learning loop work (see
*Learning loop* below).

**Seeded rows on migration:** one `Cash` source
(`kind=CASH, displayName="Cash", last4=NULL, sourceAppHint=NULL,
isUserNamed=0`). Manual entries default to this row.

**Default-Cash resolution:** `AddManualViewModel`, the backfill action,
and classifier rule 7 all resolve "the Cash source" by querying
`SELECT id FROM funding_sources WHERE kind = 'CASH' ORDER BY id ASC
LIMIT 1`. The migration inserts exactly one Cash row, so this is
deterministic on fresh installs. If the user later creates additional
Cash rows in Settings (for cash-MYR vs cash-USD, say), the original
seeded row stays the default because it has the lowest id. The user
can still pick a different Cash row in the manual-entry picker per
transaction.

### `Transaction` gets a new column

```kotlin
val fundingSourceId: Long? = null
```

Foreign key with `ON DELETE SET NULL` matching the existing
`categoryId` pattern. Nullable because:

- existing rows on upgrade have NULL until the user runs the backfill
  action
- notifications with no detectable funding hints stay NULL (user picks
  manually in the edit sheet)

New index `idx_transactions_funding_source` on `fundingSourceId`.

### Learning loop

The classifier looks up an existing FundingSource by
`(sourceAppHint, last4)` only. When it finds one, it **reuses the row
as-is and never overwrites `kind`, `displayName`, or any other
field.** Only when no row matches does the classifier insert a new
one with its inferred kind.

This means: if the classifier guesses `CREDIT_CARD` from a card-bullets
notification and the user later flips that FundingSource to
`DEBIT_BANK` in Settings, **all future notifications matching the same
`(sourceAppHint, last4)` reuse the user's correction** — because they
hit the existing row before any inference runs.

**Invariant to enforce in code:** the classifier reads
`funding_sources` but only writes when inserting a brand-new row.
Updates to existing rows come exclusively from user actions in Settings
or the edit sheet. Same pattern as the existing `merchantUserEdited`
flag for the re-parse-merchants action.

### Edge — same last-4 for both card and account at one bank

Rare but possible (HSBC card 1234 and HSBC account 1234). With `kind`
out of the unique key these collide on one row. Mitigation: user
creates a second FundingSource via Settings and re-links the affected
transactions. Not worth the schema complication for the rare case.

## Classifier

New class `FundingSourceClassifier @Inject constructor(fundingSourceDao)`
in the `parsing` package, alongside `HeuristicExtractor`.

```kotlin
suspend fun classify(
    rawText: String?,         // null for manual entries
    sourceApp: String,        // package name or MANUAL_SOURCE_APP
    now: Instant,
): Long                       // returns FundingSource.id
```

### Detection rules (first match wins)

Patterns are case-insensitive.

| order | pattern                                                                                | inferred kind | display-name template |
|-------|----------------------------------------------------------------------------------------|---------------|-----------------------|
| 1     | `with\s+(?<card>.+?)\s+[•*]+\s*(?<last4>\d{4})` (reuse `CARD_SPEND_PATTERN`'s groups) | `CREDIT_CARD` | `{card-name-verbatim} {last4}` — e.g. `"CIMB Cash Rebate Plat MasterCard 1868"` |
| 2     | `[•*]+\s*(?<last4>\d{4})` (bullets without a card-name preamble)                       | `CREDIT_CARD` | `{bank-label} card {last4}` — e.g. `"HSBC card 1868"` |
| 3     | `card\s+(?:ending\s+)?(?<last4>\d{4})`                                                 | `CREDIT_CARD` | `{bank-label} card {last4}` |
| 4     | `(?:A/C\|account)\s+(?:ending\s+)?(?<last4>\d{4})`                                     | `DEBIT_BANK`  | `{bank-label} account {last4}` |
| 5     | none of 1–4, but text contains `DuitNow`, `FPX`, `Transfer`, or `via Mobile Banking`   | `DEBIT_BANK`  | `{bank-label} (unknown account)` |
| 6     | `sourceApp ∈ {TnG, Boost, ShopeePay, Grab, Wise}` and rule 1 didn't fire               | `EWALLET`     | `{bank-label}` — e.g. `"TnG"`, `"Boost"` |
| 7     | `sourceApp == MANUAL_SOURCE_APP`                                                       | `CASH`        | seeded `"Cash"` source |
| 8     | catch-all                                                                              | `DEBIT_BANK`  | `{bank-label} (unknown account)` |

`{bank-label}` uses the existing `SourceLabels.label(sourceApp)`.
`{card-name-verbatim}` is the regex's `<card>` group with surrounding
whitespace trimmed but otherwise untouched — we do **not** strip
tokens like `MasterCard` / `Visa` / `Platinum`, because those
distinguish the user's actual different cards. User can rename in
Settings.

Rule 1's reuse of `CARD_SPEND_PATTERN`'s `<card>` group is how GWallet
and GPay pass-through works for free — the linked card name appears in
the wallet notification, so the funding source becomes the underlying
card, not the wallet.

### Lookup-then-create flow

```kotlin
val (last4, sourceAppHint, kind, displayName) = detect(rawText, sourceApp)
fundingSourceDao.findBySourceAppAndLast4(sourceAppHint, last4)?.let { return it.id }
return fundingSourceDao.insert(
    FundingSource(
        kind = kind,
        displayName = displayName,
        last4 = last4,
        sourceAppHint = sourceAppHint,
        isUserNamed = false,
        createdAt = now,
        updatedAt = now,
    ),
)
```

Wallet sources collapse on `(sourceAppHint, last4=null)` so the second
TnG tx reuses the first TnG source. Manual entries collapse on the
seeded Cash row.

The classifier works independently of `merchantRaw`. The HSBC
UNDEFINED-merchant SMS (`"Debited your A/C ending  0025 …"`) still
gets a real funding source via rule 4 → `"HSBC account 0025"`
(`DEBIT_BANK`).

## Ingest integration

In `TxIngestor`, after `HeuristicExtractor.extract` produces a
`ParsedTransaction` and before the row is inserted:

```kotlin
val fundingSourceId = fundingSourceClassifier.classify(
    rawText = parsed.rawText,
    sourceApp = parsed.sourceApp,
    now = postedAt,
)
```

The id is written into the new `Transaction.fundingSourceId` column.
For steady-state operation this is a single indexed lookup per ingest;
the insert only fires the first time a card/account/wallet identity
is seen.

`AddManualViewModel` defaults `fundingSourceId` to the seeded Cash
source id. The Add Manual sheet also exposes the funding-source picker
(see *UI*) so the user can change it inline. Picking `None` clears the
FK back to NULL.

### Cross-source dedupe

Unchanged. Dedupe key is still amount + occurredAt + merchant. When
two notifications for the same payment collapse, the surviving row
keeps the FundingSource of whichever was inserted first. This is
intentional: GWallet pushes typically arrive seconds before the bank
SMS, so the underlying card identity (from rule 1's `<card>` group)
lands first.

## Re-classification

Three triggers, only one of which is automatic.

1. **Hot path on every new ingest** (above) — automatic.
2. **Schema migration v9 → v10** — adds the column with default NULL.
   No backfill in the migration itself; the Cash source is seeded
   inside the migration.
3. **Settings → "Classify existing transactions"** — new button in the
   existing data-maintenance section, next to "Re-parse merchants."
   Iterates every row with `fundingSourceId IS NULL AND rawText IS NOT
   NULL`, runs the classifier, writes the FK. Manual entries
   (`rawText IS NULL`) get linked to the Cash source in the same pass.
   Shows progress; idempotent; reports `X of Y rows linked` when done.

User-touched FundingSources are sticky against the backfill because the
classifier never updates existing rows.

## UI

### Settings → Funding Sources screen

New entry under Settings, modeled after the existing
Descriptions/Categories management screens.

- Sectioned list grouped by kind: Credit Card / E-Wallet / Debit / Cash.
- Each row: display name, tx count for the current period, and a small
  `auto` tag when `isUserNamed = false`.
- Tap a row → bottom sheet with:
  - **Rename** (free text). Saving flips `isUserNamed = true`.
  - **Change bucket** (radio buttons for the four kinds).
  - **Merge into another source** (picker — moves all
    `fundingSourceId` references and deletes the merged row in a
    single DB transaction). Needed because the auto-detector will
    produce duplicates when the same card emits multiple notification
    shapes from different apps.
  - **Delete** (only enabled when tx count is 0).
- The seeded `Cash` row is non-deletable and its kind is locked to
  `CASH` (the bucket selector is disabled for that row). Rename is
  allowed. This prevents accidentally turning the default cash row
  into a bucket that auto-attracts notification-sourced txns. Users
  who want an additional cash-like bucket can create a fresh
  FundingSource via the Settings screen.

### Edit transaction sheet

New picker row labelled **Funding source**, positioned between the
existing Merchant and Category rows. Same UI pattern as the Category
picker — tap opens a bottom-sheet list of all FundingSources grouped
by kind, with a search field and a `None` option that clears the FK.

No "create new from picker" flow in v1 — too easy to typo-create
duplicates. New sources are created either by the classifier on ingest
or by the user in the Settings screen.

Direct user-edit on a tx writes `fundingSourceId` only; it does NOT
mutate the FundingSource row itself. (Use Settings to change kind /
rename / merge.)

### Home screen filter chip row

New chip row under the existing source/category filters:
`All / Credit / E-Wallet / Debit / Cash`. Filters at the **bucket
level only** — different cards in the same bucket collapse into one
filter selection, per the same direction that drives the CSV column.
Multi-select like the existing category chips. State persists in
`HomeViewModel` alongside the other filter states.

### Add Manual sheet

Gains the same funding-source picker row as the edit sheet, with the
seeded Cash source pre-selected.

### CSV export

New column `Source`, placed between `Source App` and `Category`. Value
is the bucket name: `Credit Card` / `E-Wallet` / `Debit/Transfer` /
`Cash`. No card identity in the CSV — identity stays in-app for
filter, rename, and the edit-sheet picker.

## Backup and cloud sync

`BackupSerialization.kt` gains a `funding_sources` array in the JSON
backup with the same latest-wins reconciliation pattern used for
categories. Match key on import is the triple `(sourceAppHint, last4,
displayName)` since ids aren't portable across devices. On conflict
between two rows that match the triple, prefer the one with
`isUserNamed = true`.

The `Transaction` JSON exports `fundingSourceLookupKey:
"<sourceAppHint>|<last4>"` (computed on export, resolved on import to
the local id) rather than the raw FK id. Same approach as the existing
`categoryName` cross-reference.

`CloudSyncWorker` adds `funding_sources` to its known-tables list and
flows changes through the existing per-table change feed.

## Testing

### JVM unit tests

- `FundingSourceClassifierTest`:
  - One test per detection rule 1–4 with realistic fixture text per
    rule.
  - Each of rules 5 (DuitNow / FPX markers), 6 (wallet sourceApp),
    7 (manual entry → Cash), 8 (catch-all → unknown account).
  - GWallet pass-through: `"… with CIMB Cash Rebate Plat MasterCard
    ••1868"` → CIMB credit card source, not GWallet.
  - HSBC UNDEFINED-merchant case (`"Debited your A/C ending  0025
    …"`) → `"HSBC account 0025"` (`DEBIT_BANK`).
  - Lookup-then-create order: first call inserts, second call with
    same `(sourceAppHint, last4)` returns the same id without
    inserting again.
  - Learning loop: existing row with user-set `kind=DEBIT_BANK` is
    returned as-is even when the classifier would infer
    `CREDIT_CARD` from the input. Mock `FundingSourceDao` with a
    pre-seeded row.
- `FundingSourceMergeTest`: repo-level test of merge — re-points FK
  references and deletes the source row in a transaction.
- `BackupSerializationTest`: round-trip a backup with a user-renamed
  funding source; assert displayName and `isUserNamed = true`
  preserved.

### Instrumented tests (androidTest — not auto-run per project rule)

- `FundingSourceDaoTest`: unique-index enforcement on `(sourceAppHint,
  last4)`, ON DELETE SET NULL propagation when a source is deleted.
- `MigrationV9ToV10Test`: schema upgrade preserves transactions,
  seeds the Cash source exactly once, leaves all `fundingSourceId`

## Migration SQL

```sql
CREATE TABLE funding_sources (
    id            INTEGER PRIMARY KEY AUTOINCREMENT NOT NULL,
    kind          TEXT NOT NULL,
    displayName   TEXT NOT NULL,
    last4         TEXT,
    sourceAppHint TEXT,
    isUserNamed   INTEGER NOT NULL DEFAULT 0,
    createdAt     INTEGER NOT NULL,
    updatedAt     INTEGER NOT NULL
);

CREATE UNIQUE INDEX idx_funding_sources_app_last4
    ON funding_sources(sourceAppHint, last4);

ALTER TABLE transactions ADD COLUMN fundingSourceId INTEGER
    REFERENCES funding_sources(id) ON DELETE SET NULL;

CREATE INDEX idx_transactions_funding_source
    ON transactions(fundingSourceId);

INSERT INTO funding_sources(
    kind, displayName, last4, sourceAppHint, isUserNamed, createdAt, updatedAt
) VALUES (
    'CASH', 'Cash', NULL, NULL, 0, <migrationTs>, <migrationTs>
);
```

`<migrationTs>` is captured at migration time
(`System.currentTimeMillis()`) and embedded as a literal in the SQL
string passed to Room.
